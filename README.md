# aissuesDB

A local MCP server that gives AI coding assistants a long-term memory for debugging. It stores hard-won bug fixes as vector-indexed entries and retrieves them via semantic search — so the same mistake is never debugged twice.

## The Problem

AI coding assistants start every session from scratch. When you spend 5+ attempts debugging a subtle framework quirk (say, closing an NSWindow from a SwiftUI button action causing EXC_BAD_ACCESS), the solution lives in your chat history and is lost by the next session. You'll hit the same bug again and watch the assistant try the same failed approaches.

Flat-file registries (`known-issues.md`) work at small scale but don't scale:
- The assistant has to read the entire file every session
- Matching is keyword-based — it can't find "my SwiftUI view won't update" when the entry says "nested ObservableObject changes don't propagate"
- As entries grow past ~50, the file bloats the context window

## The Solution

aissuesDB stores each known issue with a vector embedding of its symptom, root cause, and tags. When the assistant encounters a bug, it describes the symptom in natural language and gets back the most semantically similar past issues — complete with root causes, working fixes, and failed approaches to avoid.

```
Claude Code ──MCP (stdio)──> aissuesDB
                                ├── search_issues  →  semantic vector search
                                ├── add_issue      →  store new known issue
                                └── list_issues    →  browse by tag
                                        │
                                SQLite + sqlite-vec
```

## Setup

### Prerequisites

- Node.js 20+
- A [Voyage AI](https://voyageai.com) API key (free tier: 200M tokens)

### Install

```bash
git clone https://github.com/ameal-dev/aissuesDB.git
cd aissuesDB
npm install
npm run build
```

### Seed (optional)

If you have an existing `~/.claude/knowledge/known-issues.md` in the expected format, import it:

```bash
mkdir -p data
VOYAGE_API_KEY=<your-key> npm run seed
```

### Connect to Claude Code

Add to `~/.claude/.mcp.json` (global) or `<project>/.mcp.json` (per-project):

```json
{
  "mcpServers": {
    "aissuesDB": {
      "command": "node",
      "args": ["/absolute/path/to/aissuesDB/dist/index.js"],
      "env": {
        "VOYAGE_API_KEY": "<your-key>"
      }
    }
  }
}
```

Restart Claude Code. The three tools will be available immediately.

## MCP Tools

### `search_issues`

Semantic search over the registry. Describe a symptom or paste an error message.

```
query: "SwiftUI view not updating when nested object changes"
→ Returns: KI entry about nested ObservableObject propagation, with root cause and Combine fix
```

### `add_issue`

Store a new known issue. Designed for bugs that took 4+ attempts to solve and are likely to recur.

```
title: "NSWindow close from SwiftUI button causes EXC_BAD_ACCESS"
symptom: "EXC_BAD_ACCESS in objc_release after closing window from button action"
root_cause: "NSHostingView teardown while button call frame is on stack"
fix: "Wrap onDismiss in DispatchQueue.main.async at the call site"
failed_approaches: ["async inside dismiss function", "orderOut instead of close"]
attempts: 5
tags: ["swiftui", "macos", "nswindow"]
source_project: "KwikGrab"
```

### `list_issues`

Browse issues, optionally filtered by technology tag.

```
tag: "swiftui"  →  all SwiftUI-related entries
tag: undefined  →  all entries
```

## How It Works

Each issue is stored with a 1024-dimensional embedding generated by [Voyage AI](https://voyageai.com) (`voyage-3-large`). The embedding captures the semantic meaning of the title, symptom, root cause, and tags combined.

Search queries are embedded the same way and matched via sqlite-vec's nearest-neighbor search — so "app crashes when I tap the close button" finds an entry about "NSWindow close from SwiftUI button action causes EXC_BAD_ACCESS" even though they share no keywords.

### Data Model

| Column | Type | Description |
|--------|------|-------------|
| `id` | TEXT | Auto-generated `KI-` prefixed ID |
| `title` | TEXT | Short descriptive title |
| `symptom` | TEXT | What the user sees or what fails |
| `root_cause` | TEXT | Why it actually happens |
| `fix` | TEXT | Working solution (supports markdown) |
| `failed_approaches` | JSON array | What was tried and didn't work |
| `attempts` | INTEGER | How many attempts before solving |
| `tags` | JSON array | Technology tags |
| `source_project` | TEXT | Where it was discovered |
| `embedding` | float[1024] | Voyage AI vector embedding |

## Tech Stack

- **TypeScript** — MCP server and all tooling
- **@modelcontextprotocol/sdk** — MCP server framework (stdio transport)
- **better-sqlite3** + **sqlite-vec** — Embedded vector database
- **Voyage AI** — Embedding generation (`voyage-3-large`, 1024 dims)
- **Zod** — Input validation for MCP tool schemas

## Project Structure

```
src/

├── db.ts             # SQLite + sqlite-vec database layer
```

## CLAUDE.md Integration

Add this to your global `~/.claude/CLAUDE.md` to make the assistant use aissuesDB automatically:

```markdown
## Knowledge System (aissuesDB)

Before debugging any issue, call `search_issues` with the error message or symptom.
If a match is found, apply the documented fix directly.

After solving a problem that took 4+ attempts (not a one-off setup mistake),
call `add_issue` to persist it for future sessions.
```
